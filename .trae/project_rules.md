# 项目工作规则

## 工作方式

### 角色分工

| 角色 | 职责 |
|------|------|
| 负责人（用户） | 提出目标和要求，检查执行效果，反馈意见 |
| 助手（本系统） | 负责规划、落地、总结、改进工作 |

### 工作流程：PDCA 循环（戴明环）

所有任务采用 PDCA 循环方式推进：

| 阶段 | 说明 | 步骤 |
|------|------|------|
| **P Plan 计划** | 明确目标、分析现状、制定计划 | 1. 分析现状，找出质量问题<br>2. 分析原因和影响因素<br>3. 找出主要影响因素<br>4. 制定措施计划（5W1H） |
| **D Do 实施** | 执行计划 | 5. 执行计划或措施 |
| **C Check 检查** | 检查执行效果 | 6. 检查计划执行效果，对比预期目标 |
| **A Action 处理** | 总结经验、改进 | 7. 总结经验，制定标准（标准化）<br>8. 提出尚未解决的问题，遗留问题转入下一轮 |

### 核心理念

1. **环环相扣**：四阶段按顺序循环进行，像车轮向前滚动
2. **大环套小环**：各层级都有 PDCA 循环，上一级是下一级的依据
3. **阶梯式上升**：每完成一个循环，质量水平和管理水平提升一步

##### 执行要求

1. 接受任务时，先进行 **P 计划** 阶段，明确目标和步骤
2. 执行过程中保持透明，定期汇报进度
3. 任务完成后进行 **C 检查** 和 **A 处理**，总结经验教训
4. 将遗留问题和改进点纳入下一轮 PDCA 循环
5. **等待任务时主动检查待办清单**：每次等待用户下一步指示时，主动检查 `/Users/zhangk/workspace/Quantitative_trading/docs/AI_COLLABORATION.md` 文件中的待办事项，若有需要处理或协助的工作，先汇报任务清单再等待用户决策

## 代码审查规则

生成或修改程序后，必须从以下维度逐一审查代码：

### 审查维度

| 维度 | 审查内容 |
|------|----------|
| **1. 逻辑正确性** | 算法实现、边界条件、信号时机、数据计算公式等 |
| **2. 安全性** | SQL注入、路径遍历、敏感信息暴露、权限控制等 |
| **3. 性能** | 批量操作、索引使用、循环优化、缓存策略等 |
| **4. 健壮性** | 异常处理、空值处理、类型转换、超时控制等 |
| **5. 可维护性** | 注释完备性、命名规范性、代码重复度、模块耦合度等 |
| **6. 合规性** | 与已有表结构一致、API调用限制、数据范围规定等 |

### 执行要求

1. 代码审查应在程序生成或修改完成后立即进行
2. 发现问题应立即修复，不得遗留到后续环节
3. 复杂逻辑需额外关注边界条件和异常场景
4. 涉及数据库操作的代码需确认表结构一致性
5. 涉及外部API调用的代码需确认符合API限制要求

## 协作流程（增强黑板模式）

方舟与量量之间通过共享文件系统 + 共享 memory 实现跨会话协作，详见 [docs/协作单.md](../docs/协作单.md)。

### 核心机制

| 组件 | 用途 | 读写方 |
|------|------|--------|
| `docs/协作单.md` | 工单持久记录（唯一真相源） | 双方读写 |
| memory/topics.md | 跨会话轻量通知（状态变更信号） | 双方追加 |
| memory/project_memory.md | 经验教训积累（知识库） | 双方读写 |

### 工单生命周期

```
NEW（提单） → ASSIGNED（认领） → VERIFY（待验证） → CLOSED（关闭）
                                              ↘ REOPENED（打回）
```

### 通知约定

每次工单状态变更，**同时**在 memory topics.md 追加一行通知对方：

```
[方舟/量量 YYYY-MM-DD HH:mm] 协作单 [ID] 状态变更: NEW→ASSIGNED（标题简述）
```

### 启动检查（强制）

每次新会话，**必须先执行启动检查，再处理任何任务。**

#### 方舟启动时
1. Read `.trae/topics.md`（跨会话通知，查看是否有新状态变更）
2. Read `docs/协作单.md` 最后 50 行检查工单最新动态
3. 如果有 `VERIFY` 且验证方是自己 → 优先验证
4. 汇报当前所有 OPEN（非 CLOSED）工单概览

#### 量量启动时
1. Read `.trae/topics.md`（跨会话通知，查看是否有新状态变更）
2. Read `docs/协作单.md` 最后 50 行检查工单最新动态
3. 如果有 `NEW` 且属于自己职责范围 → 优先认领
4. 如果有 `REOPENED` 且是自己上一次处理的 → 优先跟进

## 对话管理系统规则

### 系统概述

对话管理系统是基于**三层机制 + Git版本化**的上下文管理工具，专门为解决长周期项目开发中的上下文窗口限制问题而设计。系统完美适配PDCA工作流，支持自动化上下文监控、高质量总结生成和新对话自动恢复。

### 核心原则

1. **PDCA融合原则**：对话管理必须与PDCA循环紧密结合
2. **版本化原则**：所有对话总结必须纳入Git版本管理
3. **自动化原则**：尽可能自动化执行上下文监控和管理

### 文件结构规则

#### 1. 核心文件位置
所有对话管理相关文件统一存放在 `.trae/` 目录下：

```
.trae/
├── conversation_manager.py      # 核心管理模块
├── summary_template.md          # 标准化摘要模板
├── reload_context.py            # 新对话自动加载脚本
├── conversation_history/        # 对话历史存储目录
│   └── summary_*.json           # 自动生成的总结文件
└── project_rules.md             # 本项目规则文件
```

### 三层对话管理机制规则

#### 1. 实时摘要层
- **触发时机**：每次重要里程碑完成后
- **执行要求**：
  - 生成轻量化摘要，不占用过多算力
  - 记录核心工作内容和关键决策
  - 更新待办事项列表

#### 2. 智能压缩层
- **触发阈值**：上下文使用率达到80%时自动触发
- **执行要求**：
  - 生成结构化高质量总结
  - 保存为标准化JSON文件
  - 自动执行Git提交

#### 3. 重启恢复层
- **触发时机**：每次新对话开始时
- **执行要求**：
  - 自动加载最新对话总结
  - 重建项目上下文状态
  - 提供接续关键词

### 执行流程规则

#### 1. 新对话启动流程
```
1. 运行：python3 .trae/reload_context.py
2. 检查加载的历史上下文
3. 确认当前项目状态
4. 继续PDCA循环
```

#### 2. 上下文监控流程
```
1. 定期估算当前token使用量
2. 检查是否达到80%阈值
3. 达到阈值时自动生成总结
4. 保存文件并Git提交
```

#### 3. 里程碑总结流程
```
1. 完成重要功能或阶段后
2. 调用总结生成函数
3. 记录已完成工作和待办事项
4. 更新项目状态信息
```
