# 协作单

> 方舟与量量之间的工单系统，基于增强黑板模式。
> 最后更新：2026-06-06

---

## 工作流

### 状态机

```
NEW（方舟/量量提单）
  │
  ▼
ASSIGNED（量量/方舟 认领并开始处理）
  │
  ▼
VERIFY（处理完成，待对方验证）
  │
  ├─ ✅ → CLOSED（验证通过，关闭）
  │
  └─ ❌ → REOPENED（验证不通过 → 回到 ASSIGNED）
```

### 各角色职责

| 角色 | 在工单中的职责 |
|------|---------------|
| **方舟** | 发现前端/后端问题 → 提单（NEW）；量量修复后做前端侧验证（VERIFY→CLOSED/REOPENED） |
| **量量** | 修复后端/数据问题（ASSIGNED→VERIFY）；方舟提的后端问题，量量认领修复 |
| **K** | 仲裁争议（REOPENED 后双方无法达成一致时）；最终验收 |

### 生命周期规则

1. **提单方**（通常是发现方）写入工单时状态为 `NEW`
2. **处理方**认领后改为 `ASSIGNED`，开始修复
3. **处理方**完成修复并验证后改为 `VERIFY`，**同时**在 topics.md 追加一行通知对方
4. **验证方**从前端/使用侧验证后：
   - 验证通过 → `CLOSED`
   - 验证不通过 → `REOPENED`，注明原因，**同时**在 topics.md 追加一行通知处理方
5. 一个工单最多 REOPENED 2 次，第 3 次需要 K 介入仲裁

## 通知约定

每次状态变更，**除了更新协作单本身**，同步在 memory topics.md 追加一行：

```
[方舟/量量 YYYY-MM-DD HH:mm] 协作单 [ID] 状态变更: NEW→ASSIGNED（标题简述）
```

对方下次启动会话时，memory 会自动加载，无需 K 中转。

## 启动检查（强制）

每次新会话，**必须先执行启动检查，再处理任何任务。**

### 方舟启动时

每次新会话，自动执行：
1. Read `.trae/topics.md`（跨会话通知，查看是否有新状态变更）
2. Read `docs/协作单.md` 最后 50 行检查工单最新动态
3. 如果有 `VERIFY` 且验证方是自己 → 优先验证
4. 汇报当前所有 OPEN（非 CLOSED）工单概览

### 量量启动时

每次新会话，自动执行：
1. Read `.trae/topics.md`（跨会话通知，查看是否有新状态变更）
2. Read `docs/协作单.md` 最后 50 行检查工单最新动态
3. 如果有 `NEW` 且属于自己职责范围 → 优先认领
4. 如果有 `REOPENED` 且是自己上一次处理的 → 优先跟进

**修复**：
1. `stocks.py:get_stocks()` — 将 `as_of_date: str = DateDep` 改为 `as_of_date: str = Query(None)`，路由内未传时自动使用 `screener.trade_date`
2. 同时重生成 parquet 文件（从 DB 导出最新 5515 条数据）
3. `DataLoader` 增加 parquet 文件 mtime 检测自动重载机制（解决 META 陈旧数据问题）
4. `ScreenerService` 的 df/trade_date/field_counts 改为 `@property` 动态读取 loader（解决服务层缓存问题）

## [4.4-ASSESS-20260610] Phase 4.4 策略持久化方案评估

| 字段 | 内容 |
|------|------|
| **状态** | CLOSED（评估即关闭，无代码改动） |
| **提单方** | 方舟 |
| **日期** | 2026-06-10 |
| **影响范围** | 无（仅决策记录） |
| **优先级** | P3（不阻塞功能） |

**背景**:
Phase 4.4 联调过程中，发现 `StrategyManager` 使用 `LocalStorageStrategyStorage`（utils/strategyStorage.ts）做策略持久化，注释 L6 提到"Phase 2 可替换为 HttpStrategyStorage"。需评估是否在本期切换为后端 API。

### 评估对比

| 维度 | 保留 localStorage | 切换为后端 API |
|------|-------------------|---------------|
| 即时响应 | ✅ 同步读，零延迟 | ❌ 需网络请求 + 等待 |
| 离线可用 | ✅ 完全支持 | ❌ 离线无法访问 |
| 跨设备同步 | ❌ 不支持（可短码导出弥补） | ✅ 支持 |
| 数据可靠性 | ⚠️ 浏览器清缓存会丢 | ✅ 服务端持久化 |
| 用户登录态 | ✅ 不依赖 | ❌ 需要 user_id 体系 |
| 后端支持 | ✅ 0 改动 | ❌ 需建 user_strategies 表 + 5 个接口 |
| 容量 | localStorage 5-10MB（已限制 50 条） | 取决于后端设计 |

### 决策

**保留 localStorage**，理由：
1. 策略属用户偏好类（不是核心交易数据），本地存储符合产品定位
2. 跨设备需求可用现有"导出短码 / JSON 文件"功能弥补（见 StrategyManager.tsx L165-172）
3. 切换后端需新建 `user_strategies` 表 + 5 个 CRUD 接口，工期 ~2 天，价值不高
4. 后端若后续需要，可平滑替换为 `HttpStrategyStorage`（接口已抽象好）

**保留内容**:
- `LocalStorageStrategyStorage`（utils/strategyStorage.ts）
- 短码/JSON 导入导出（已支持跨设备分享）
- 50 条上限（容量保护）

**处理记录**:
- 2026-06-10 方舟: 评估完成（CLOSED），无需代码改动
