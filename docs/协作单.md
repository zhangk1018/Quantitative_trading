## [6.10-CYCLE-NAMING-20260617] 北交所新股 stock_quotes cycle 命名不一致导致指标计算缺失

| 字段 | 内容 |
|------|------|
| **状态** | **WITHDRAWN**（经核验：get_quotes 有 cycle_map={'daily':'1d'}，77只NaN是次新股数据<60天被门槛跳过，非cycle命名问题） |

| **日期** | 2026-06-17 |
| **影响范围** | backend |
| **优先级** | P2 |

**验收方式**:

## 工作流

### 状态机

```
NEW（方舟/量量提单）
  │
  ▼
ASSIGNED（量量/方舟 认领并开始处理）
  │
  ▼
VERIFY（处理完成，待对方验证）
  │
  ├─ ✅ → CLOSED（验证通过，关闭）
  │
  └─ ❌ → REOPENED（验证不通过 → 回到 ASSIGNED）
```

### 各角色职责

| 角色 | 在工单中的职责 |
|------|---------------|
| **方舟** | 发现前端/后端问题 → 提单（NEW）；量量修复后做前端侧验证（VERIFY→CLOSED/REOPENED） |
| **量量** | 修复后端/数据问题（ASSIGNED→VERIFY）；方舟提的后端问题，量量认领修复 |
| **K** | 仲裁争议（REOPENED 后双方无法达成一致时）；最终验收 |

### 生命周期规则

1. **提单方**（通常是发现方）写入工单时状态为 `NEW`
2. **处理方**认领后改为 `ASSIGNED`，开始修复
3. **处理方**完成修复并验证后改为 `VERIFY`，**同时**在 topics.md 追加一行通知对方
4. **验证方**从前端/使用侧验证后：
   - 验证通过 → `CLOSED`
   - 验证不通过 → `REOPENED`，注明原因，**同时**在 topics.md 追加一行通知处理方
5. 一个工单最多 REOPENED 2 次，第 3 次需要 K 介入仲裁

### 评估对比

| 维度 | 保留 localStorage | 切换为后端 API |
|------|-------------------|---------------|
| 即时响应 | ✅ 同步读，零延迟 | ❌ 需网络请求 + 等待 |

| 容量 | localStorage 5-10MB（已限制 50 条） | 取决于后端设计 |

## 第三步技术评审报告：前端 fetch 异常分类 + 暂缓节点

### 问题描述
协作单 [6.11] 原写"前端 axios 把 ECONNREFUSED 包装为 500 错误"——**经核查前端实际使用原生 `fetch` 而非 axios**（`frontend/src/api.ts` 全局用 `fetch` + `AbortController`）。实际错误链路：
- 浏览器发起 `GET /api/stocks/` → vite proxy :5173 → :8000
- :8000 无服务（Docker daemon 未运行）→ TCP RST → fetch 抛 `TypeError: Failed to fetch`
- 业务层 `if (!res.ok) throw new Error(...)` 不会执行（因为 fetch 失败不返回 res）
- UI 错误提示："Request failed with status code 500"（浏览器 Network 面板渲染 vite proxy 的 502 状态）

### 技术方案

**最小修改（建议 V2.0 实施）**：
```typescript
// 1. 新增 ApiError 类（frontend/src/types.ts）
export class ApiError extends Error {
  constructor(
    public type: 'network' | 'http',
    public status: number,
    message: string
  ) {
    super(message)
    this.name = 'ApiError'
  }
}

// 2. 统一 fetch 封装（frontend/src/api.ts）
async function apiFetch(url: string, init?: RequestInit): Promise<Response> {
  try {
    return await fetch(url, init)
  } catch (e) {
    if (e instanceof TypeError) {
      // 浏览器 fetch 网络层失败（ECONNREFUSED / DNS / CORS 等）
      throw new ApiError('network', 0, `网络连接失败：${e.message}`)
    }
    throw e
  }
}

// 3. 调用方替换（fetchStocks / fetchKline / fetchWatchlist 等）
const res = await apiFetch(`${BASE}/stocks/?${p}`, { signal: config?.signal })
if (!res.ok) throw new ApiError('http', res.status, `/api/stocks/ failed: ${res.status}`)

// 4. UI 层差异化提示
if (e.type === 'network') toast.error('后端服务未启动，请联系管理员或运行 ./deploy.sh')
else if (e.status === 502) toast.error('后端启动中，请稍候...')
else toast.error(`请求失败 (${e.status})`)
```

**影响面**：
- 5 个 fetch 函数（fetchMeta / fetchStocks / fetchKline / fetchSignals / fetchWatchlist / addWatchlist / deleteWatchlist / updateWatchlist / fetchStockByCode）
- UI 层 4 个调用点（StockPickerView / StockDetail / Watchlist / Monitor）
- 单测：5+ 用例覆盖 network/http 分类

## [9.1-ADJ-FACTOR-BUG-20260719] stock_adj_factor 表复权因子数据全量错误——存的是后复权因子(turn)而非前复权因子(adjfactor)

| 字段 | 内容 |
|------|------|
| **状态** | IN_PROGRESS → 待重导 stock_quotes |
| **提单方** | 方舟 |
| **处理方** | 量量 |
| **日期** | 2026-07-19 |
| **影响范围** | backend（全市场所有股票前复权K线价格错误） |
| **优先级** | **P0**（核心数据错误，前台K线价格与真实价格偏差可达数倍） |

**问题现象**:
股票 **300795 米奥会展**，数据库 `stock_quotes` 中 2026-04-30 原始收盘价 10.89 元，但前台 K 线图（前复权）显示 3.31 元，偏差 3.3 倍。K 反馈该公司仅做了 10 派 2 元现金分红，**未送转股**，前复权价不应出现如此巨大差异。

**根因分析**:
`stock_adj_factor` 表存储的 **不是** Baostock 的 `foreAdjustFactor`（前复权因子，即 `adjfactor`），而是 **错误地** 存储了 `backAdjustFactor`（后复权因子，即 `turn`）。

**证据链**:
1. 直接查询 Baostock API 验证字段：
   - `query_adjust_factor(code='sz.300795')` 返回字段顺序：
     - `row[0]` = code（如 `sz.300795`）
     - `row[1]` = dividOperateDate（除权除息日，如 `2026-04-30`）
     - `row[2]` = **foreAdjustFactor**（前复权因子，2026-04-30 值为 `1.000000`）
     - `row[3]` = **backAdjustFactor**（后复权因子/turn，2026-04-30 值为 `3.289880`）
     - `row[4]` = adjustFactor（与 backAdjustFactor 相同值）
2. 数据库中 300795 的最新因子 `3.2900 ≈ 3.289880 = turn`（后复权因子），**不是** `1.000000 = adjfactor`（前复权因子）。
3. `backend/imputer/adjuster.py` 前复权公式：`ratio = adj_factor / latest_factor`。若 `latest_factor = 3.29`（后复权因子，远大于1），则 4 月 30 日 `ratio = 1.0 / 3.29 ≈ 0.304`，前复权价 = `10.89 × 0.304 ≈ 3.31` 元——与前台显示完全吻合。

**错误计算过程**（以 300795 为例）：
```
正确情况（应该的）：
  4月30日 adjfactor = 1.0，最新 adjfactor = 1.0
  前复权价 = 10.89 × (1.0/1.0) = 10.89 元 ✅

错误情况（当前数据库）：
  4月30日 被存为 turn = 3.2259，最新被存为 turn = 3.2899

  ratio = 1.0/3.29 = 0.304，前复权价 = 10.89 × 0.304 = 3.31 ❌
```
