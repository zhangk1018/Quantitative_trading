## 二、五大核心风险点与应对方案

---

### 🔴 风险点 1：时间戳严格对齐问题 (最常见 Bug)

#### 现象
买卖点标记没有显示，或者显示在了错误的 K 线上。

#### 原因
后端返回的 K 线时间戳与回测引擎记录的买卖点时间戳存在微小差异（如相差几毫秒，或一个是 UTC 时间，一个是本地时间）。`klinecharts` 的覆盖物匹配是**严格相等**的。

#### ✅ 应对方案
1. **后端 API 统一返回 **Unix 毫秒时间戳** 或标准化的字符串（如 `YYYY-MM-DD HH:mm:ss`）**
2. **前端渲染前，写映射函数，确保 `signal.timestamp === kline.timestamp`
3. **微小误差处理：使用容差匹配 `Math.abs(signal.time - kline.time) < 60000`（1分钟容差）

---

### 🔴 风险点 2：复权价格与买卖点价格错位

#### 现象
经历过分红、拆股的股票，买卖点标记悬空在 K 线上方或下方很远的地方。

#### 原因
前端 K 线图展示的是"前复权"价格，但回测引擎传给前端的买卖点 `price` 是"不复权"的原始成交价。

#### ✅ 应对方案
**建立铁律**：
- 前端展示的 K 线数据
- 回测引擎计算的逻辑价格
- 最终传给前端的买卖点价格

**必须全部是同一套复权标准（强烈推荐统一使用"前复权"）**

---

### 🔴 风险点 3：密集信号导致的 UI 重叠与遮挡

#### 现象
在震荡市中，策略频繁发出买卖信号，多个三角形标记重叠在一起，甚至遮挡了 K 线实体，导致图表无法阅读。

#### ✅ 应对方案
1. **数据层过滤**：
   - 同一天（或同一根 K 线）内如果有多个同向信号，只保留最后一个
   - 如果一买一卖，只保留净结果
2. **渲染层偏移**：
   - 利用 `klinecharts` 覆盖物的 `y` 轴偏移属性
   - 根据该 K 线的高低点动态计算 Y 坐标，例如：
     ```javascript
     y = high + (high - low) * 0.1
     ```
   - 确保标记永远在 K 线实体之外

---

### 🔴 风险点 4：跨周期 (Timeframe) 数据不一致

#### 现象
用户在回测时使用的是 15 分钟级别数据，但前端图表默认加载了日线数据，导致买卖点完全对不上。

#### ✅ 应对方案
1. **前端 API 请求必须携带 `frequency` 参数（如 `?symbol=000001&freq=15m`）
2. **后端返回的 K 线和 Signal 数据必须严格基于该周期
3. **前端切换周期按钮时，必须清空当前图表并重新请求

---

### 🔴 风险点 5：频繁切换股票导致的前端内存泄漏

#### 现象
用户在选股器中快速点击不同的股票查看 K 线，浏览器逐渐卡顿甚至崩溃。

#### ✅ 应对方案
在 React 的 `useEffect` 清理函数（cleanup function）中：
1. 调用 `klinecharts` 的销毁方法（如 `chart.dispose()` 或 `chart.clearData()`）
2. 清理前端按 `stock_code` 缓存的旧数据

## 三、前后端 API 契约设计建议

### `/api/signals` 返回的标准 JSON 结构

```json
{
  "symbol": "000001",
  "frequency": "1d",
  "klines": [
    {
      "timestamp": 1690000000000,
      "open": 150,
      "high": 155,
      "low": 149,
      "close": 153,
      "volume": 10000
    }
  ],
  "trades": [
    {
      "trade_id": "T001",
      "timestamp": 1690000000000,
      "side": "BUY",
      "price": 153.5,
      "volume": 100,
      "pnl": null,
      "is_profit": null
    },
    {
      "trade_id": "T002",
      "timestamp": 1695000000000,
      "side": "SELL",
      "price": 160.0,
      "volume": 100,
      "pnl": 650.0,
      "is_profit": true
    }
  ]
}
```

**设计说明：
- `side: "BUY" / "SELL"
- `is_profit`: 平仓时标记盈亏状态
- `pnl`: 平仓时计算盈亏金额
- 前端拿到数据后，可以直接遍历 `trades` 数组渲染

## 四、开发注意事项

### 防未来函数验证流程
在量化开发中，"看不见摸不着"的回测数字是不可信的，只有将买卖点精准地画在 K 线上，才能一眼识破"未来函数"、"滑点设置不合理"或"撮合逻辑错误"。

### 技术选型说明
前端使用 `klinecharts` (基于 Canvas) 在处理千根以上 K 线和复杂覆盖物时的性能，远超基于 SVG 的库（如 Plotly 或 Echarts）。
