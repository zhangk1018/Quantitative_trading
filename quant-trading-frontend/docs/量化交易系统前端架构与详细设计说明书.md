## 前言：设计理念与重构原则

本方案基于 **“全新设计、零历史包袱”** 的原则，彻底摒弃原有代码的耦合结构，采用当前前端工程化的**最优实践（Best Practices）** 进行从零构建。

**核心设计哲学：**
1. **高内聚，低耦合**：每个组件只负责单一职责，视图（View）、状态（Store）、请求（Service）严格分离。
2.  **数据驱动 UI**：彻底摒弃 DOM 操作思维，所有界面变化均由状态层（V1.0 采用 `useReducer` + `ScreenerContext`；V2.0 计划升级为 Zustand）或 TanStack Query 数据驱动。
3.  **极致性能**：针对金融数据"高频、大数据量"的特点，在图表渲染、表格滚动、状态更新层面做深度优化。

> **V1.0 实际落地差异说明**（2026-06 更新）：因 V1.0 阶段业务规模适中、团队学习成本考量，状态管理采用 `React.useReducer` + `ScreenerContext`（V2.0 文档原规划的 Zustand）；数据请求层暂未引入 TanStack Query，使用 `fetch` + `useEffect` 自封装 hooks（已具备缓存 / 取消 / Loading 规范）。V2.0 阶段将按本文档原方案升级为 Zustand + TanStack Query。**条件构建器（含自编指标）** V1.0 已完整落地，P1-P6 任务全部闭环，详见 [自编指标章节](#九条件构建器--自编指标-v10-2026-06-完整实施)。

## 一、 系统总体架构设计

### 1.1 工程目录结构（企业级标准）

采用 **“按业务领域（Domain）划分”** 的目录结构，而非传统的“按文件类型划分”，这能极大提升中大型项目的可维护性。

### 1.2 技术栈落地规范

| 技术层 | 选型 | 落地规范与约束 |
| :--- | :--- | :--- |
| **基础框架** | React 18 + TS 5.x | 严格开启 `strict: true`，禁止使用 `any`，所有组件必须定义 Props 接口。 |
| **UI 组件库** | Ant Design 5.x | 使用 CSS-in-JS 的 `ConfigProvider` 统一配置暗色主题，禁止直接修改 AntD 内部 CSS。 |
| **样式方案** | Tailwind CSS 3.x | 用于 90% 的布局、间距、颜色控制；仅对极复杂的自定义动画使用 CSS Modules。 |
| **状态管理** | Zustand 4.x | 采用 `immer` 中间件处理复杂对象更新；Store 按业务模块拆分，禁止单一巨型 Store。 |
| **数据请求** | TanStack Query | 所有异步数据获取必须通过 `useQuery` / `useMutation`，禁止在 `useEffect` 中手写 `fetch`。 |
| **图表引擎** | Lightweight Charts | 封装为独立的 React 组件，通过 `ref` 和 `useEffect` 管理生命周期，避免重复渲染。 |

## 二、 全局设计与规范

### 2.1 暗色主题与视觉规范 (Design Tokens)

金融终端的核心是“长时间注视不疲劳”与“数据高对比度”。我们将通过 Ant Design 5 的 Token 机制和 Tailwind 配置来实现全局统一。

**核心色板定义：**
*   **背景层级**：
    *   `bg-base` (`#131722`): 全局最底层背景。
    *   `bg-panel` (`#1E222D`): 侧边栏、顶部栏、弹窗背景。
    *   `bg-card` (`#2A2E39`): 内容区卡片、表格行背景。
*   **文字层级**：
    *   `text-primary` (`#EAECEF`): 核心数据、标题。
    *   `text-secondary` (`#848E9C`): 标签、辅助说明。
    *   `text-disabled` (`#5E6673`): 禁用状态。
*   **金融语义色**：
    *   `color-up` (`#26A69A`): 上涨、买入信号、正向收益（国际惯例绿涨）。
    *   `color-down` (`#EF5350`): 下跌、卖出信号、负向亏损。
    *   `color-accent` (`#2962FF`): 品牌主色、选中状态、主要操作按钮。

### 2.2 全局布局架构 (App Layout)

采用经典的 **“T 型布局”**，确保核心工作区最大化。

1.  **顶部状态栏 (Header, 56px)**：
    *   **左侧**：系统 Logo + 名称。
    *   **右侧**：系统状态指示灯（带 Tooltip）、策略运行状态、最后数据更新时间、用户头像/帮助菜单。
    *   *设计 rationale*：将全局状态置于顶部，用户在任何子视图都能感知系统健康度。
2.  **左侧导航栏 (Sidebar, 240px)**：
    *   采用 Ant Design `Menu` 组件，垂直排列四大核心视图（选股、自选、回测、配置）。
    *   支持折叠（Collapse）为 64px 图标模式，为图表区腾出空间。
3.  **主工作区 (Main Content, Flex: 1)**：
    *   根据路由动态渲染对应的 Feature 模块。
    *   内部采用 `Flex` 布局，分为“左侧控制/筛选区 (固定宽度)”和“右侧数据/图表区 (自适应宽度)”。

## 三、 核心视图详细设计

### 3.1 选股视图 (Stock Picker) - 最复杂的业务模块

**设计目标**：将复杂的量化因子筛选转化为直观的“搭积木”体验。

#### 3.1.1 左侧筛选控制区 (Filter Control Panel)
*   **模块划分**：
    1.  **标的池过滤**：市场、板块、排除 ST/停牌/次新（使用 AntD `Checkbox.Group` 和 `Select`）。
    2.  **因子条件构建器 (Condition Builder)**：
        *   *创新设计*：不使用传统的表单，而是采用"卡片流"设计。每添加一个条件，生成一个可拖拽、可删除的 `ConditionCard`。
        *   卡片内部包含：指标选择（如 PE）、运算符（<, >, =）、数值输入。
        *   卡片之间通过 `LogicConnector` (AND/OR) 连接。
        *   **V1.0 落地**（2026-06）：实际采用 **扁平无嵌套** 数据结构（`FilterTree.conditions[]` 有序列表），每个 `FilterCondition` 独立携带 `op: 'AND' | 'OR' | 'NOT'`。3 个互斥单选按钮控制"下一个待添加"的关系；op 标签可点击循环切换 `AND → OR → NOT → AND`。**已支持自编指标**（系统预设 + 我的自编双 Tab 切换），详见 [条件构建器 + 自编指标 V1.0 实施章节](#九条件构建器--自编指标-v10-2026-06-完整实施)。
    3.  **因子打分配置 (Factor Scoring)**：
        *   使用 AntD `Slider` 组件，支持多滑块联动，总和必须为 100%（通过自定义校验实现）。
*   **底部操作区**：固定悬浮的“开始选股”主按钮（Primary）和“重置”次按钮（Default）。

#### 3.1.2 右侧数据展示区 (Data Display Area)
*   **顶部工具栏**：
    *   排序与 TopN 控制。
    *   **策略管理入口**：“保存策略”与“我的策略”按钮。
*   **核心表格 (Stock Table)**：
    *   使用 AntD `Table` 组件，采用**前端内存分页**（每页 50 条，默认 50 条/页），不使用虚拟滚动（`virtual`）。结果集通常在几百条以内，原生分页完全足够，避免虚拟滚动带来的配置复杂度。
    *   **自定义列渲染**：涨跌幅、收益率等列，根据正负值自动应用 `text-up` 或 `text-down` 样式。
    *   **行选择**：支持单选/多选，选中行高亮，并联动底部操作栏。
*   **底部批量操作栏**：当表格有选中项时，从底部滑出（Slide-up）操作栏，包含“加入回测”、“添加自选”、“导出”等。

#### 3.1.3 策略管理抽屉 (Strategy Drawer)
*   点击“我的策略”从右侧滑出 AntD `Drawer`。
*   **列表展示**：每个策略为一个 `Card`，显示名称、更新时间、包含的条件数量摘要。
*   **操作交互**：
    *   **加载**：一键将策略条件恢复到左侧筛选区，并自动触发选股。
    *   **导入/导出**：支持 JSON 文件上传/下载，以及 Base64 短码的复制/粘贴（通过 Modal 实现）。

### 3.2 回测视图 (Backtest) - 图表与性能核心

**设计目标**：提供专业级的 K 线交互体验，并清晰展示回测绩效。

#### 3.2.1 左侧回测控制区
*   **标的管理**：支持多标的切换，使用 `Radio.Group` 或自定义的 `StockTag` 列表。
*   **周期与复权**：使用 AntD `Segmented` (分段控制器) 实现周期切换（日/周/月/分钟），体验优于传统 Select。
*   **图表图层控制**：使用 `Switch` 和 `Checkbox` 控制 MA、MACD、买卖信号的显示与隐藏。

#### 3.2.2 右侧图表引擎区 (Chart Engine) - 深度集成 Lightweight Charts
这是整个系统的技术难点，我们将采用 **“主图 + 多副图同步联动”** 的架构。

*   **图表容器设计**：
    *   采用 Flex 纵向布局，主图占比 60%，副图1（成交量）占比 20%，副图2（MACD/RSI）占比 20%。
    *   **核心机制**：三个图表实例共享同一个 `TimeScale` (时间轴)。通过 Lightweight Charts 的 `syncWith` 或自定义的 `crosshairMove` 事件监听，实现十字光标在三个图表间的**绝对同步**。
*   **K 线与指标渲染**：
    *   **主图**：`CandlestickSeries` (K线) + 多个 `LineSeries` (MA5/10/20/60，颜色按设计规范配置)。
    *   **买卖信号 (Markers)**：使用 `setMarkers` API。买入信号为绿色向上三角（位于最低价下方），卖出信号为红色向下三角（位于最高价上方）。点击 Marker 可触发 React 弹窗显示交易详情。
*   **性能优化**：
    *   图表数据更新时，使用 `setData` 而非 `update` 进行全量替换（针对回测场景）。
    *   开启 WebGL 加速（如果 LC 版本支持），确保 60fps 流畅度。

#### 3.2.3 绩效分析面板 (Performance Panel)
*   位于图表下方，采用 CSS Grid 布局。
*   **指标卡片**：总收益、年化收益、最大回撤、夏普比率等。
    *   *视觉设计*：卡片背景使用 `bg-card`，数值使用大字号、加粗，并根据正负值着色（如最大回撤始终为红色）。
*   **交易明细表**：展示每一笔买卖的日期、方向、价格、盈亏。支持按盈亏排序，快速定位“神操作”或“神坑”。

### 3.3 自选股视图 (Watchlist)
*   **设计思路**：轻量级、高频查看。
*   **左侧分组树**：使用 AntD `Tree` 组件管理“全部”、“金融”、“科技”等自定义分组。
*   **右侧行情表**：
    *   除了常规的价格、涨跌幅，增加“迷你走势图 (Sparkline)”。
    *   *技术实现*：迷你走势图（Sparkline）**必须使用纯 SVG 绘制**（如 `d3-shape` 的 line 生成器或原生 SVG path），仅渲染收盘价折线路径，不带任何坐标轴和交互。**严禁在 AntD Table 单元格内实例化 Lightweight Charts**——一页 50 条数据会实例化 50 个 WebGL 图表，导致 WebGL 上下文丢失和页面崩溃。

### 3.4 参数配置视图 (Configuration)
*   **设计思路**：严谨、防错。
*   **布局**：左侧为配置分类菜单（账户、费率、风控等），右侧为对应的表单区域。
*   **表单设计**：
    *   使用 AntD `Form` 进行全局表单管理。
    *   **输入校验**：所有数值输入框（如佣金、仓位）必须配置 `min`、`max` 和自定义 `validator`（如：单笔仓位 + 其他仓位不能超过 100%）。
    *   **脏检查 (Dirty Check)**：当用户修改了配置但未保存时，切换左侧菜单或视图时，弹出 `Modal.confirm` 提示“配置未保存，是否离开？”。

## 四、 数据流与状态管理设计

采用 **“分层状态管理”** 策略，彻底解决数据流混乱问题。

### 4.1 Zustand (客户端状态 - 存“配置”与“UI状态”)
Zustand 仅用于存储**不需要频繁向服务器请求、且需要跨组件共享**的数据。

*   **`useAppStore`**：当前路由、侧边栏折叠状态、全局 Loading。
*   **`useStockPickerStore`**：当前的筛选条件对象、因子权重、选中的股票 ID 列表。
*   **`useBacktestStore`**：当前选中的标的代码、K 线周期、复权方式、图表图层显示开关。
*   **`useConfigStore`**：账户资金、费率设置、风控阈值（支持从 localStorage 持久化）。

### 4.2 TanStack Query (服务端状态 - 管“异步数据”)
所有与“数据获取”相关的逻辑，全部交由 TanStack Query 接管。

*   **优势利用**：
    *   **自动缓存与去重**：在选股视图和自选股视图中，如果请求了同一只股票的数据，TanStack Query 会自动复用缓存，避免重复请求。
    *   **后台静默更新 (Background Refetch)**：设置 `refetchInterval`，在自选股视图中每隔 30 秒静默刷新行情，UI 无感更新。
*   **乐观更新 (Optimistic Updates)**：在“删除自选股”或“保存策略”时，先立即更新 UI，再发送请求，失败则回滚，提供极速体验。
    *   *强制规范*：所有使用乐观更新的 `useMutation` **必须配套 `onError` 回滚逻辑**（回退缓存数据并展示错误提示），并在 `onSettled` 中调用 `queryClient.invalidateQueries()` 确保与服务端最终一致性。严禁只写乐观更新而无回滚机制。

### 4.3 数据流转示例：执行一次“选股”
1.  用户在左侧修改筛选条件 -> 更新 `useStockPickerStore` 中的 `filters` 状态。
2.  用户点击“开始选股” -> 触发 TanStack Query 的 `refetch()`。
3.  TanStack Query 将 `filters` 作为参数，调用后端 API（或 Mock API）。
4.  请求期间，UI 显示骨架屏（Skeleton）。
5.  请求成功，数据存入 TanStack Query 缓存，并触发 UI 重新渲染，表格展示结果。

## 五、 实施计划与里程碑 (Roadmap)

项目预计总工期为 **6 周**，分为 5 个迭代周期（Sprint）。

### Sprint 1: 基建与骨架 (Week 1)
*   **目标**：搭建工程，跑通路由，实现全局暗色主题和基础布局。
*   **任务**：
    1.  初始化 Vite + React + TS 项目，配置 ESLint/Prettier。
    2.  配置 Tailwind CSS 和 Ant Design 5 的暗色主题 Token。
    3.  实现 `AppLayout`（Header + Sidebar + Outlet）。
    4.  配置 React Router，实现四大视图的空壳页面切换。

### Sprint 2: 选股视图与策略管理 (Week 2)
*   **目标**：完成最复杂的选股交互，实现策略的本地持久化。
*   **任务**：
    1.  开发左侧 `FilterPanel`（条件构建器、因子滑块）。
    2.  开发右侧 `StockTable`（集成虚拟滚动、自定义列渲染）。
    3.  实现 `StrategyManager`（抽屉、列表、增删改查、JSON 导入导出）。
    4.  打通 Zustand 状态，实现条件修改与表格数据的联动（使用 Mock 数据）。

### Sprint 3: 回测视图与图表引擎 (Week 3)
*   **目标**：攻克 Lightweight Charts 集成，实现专业的 K 线展示。
*   **任务**：
    1.  封装 `useLightweightChart` Hook（处理生命周期、Resize、主题）。
    2.  实现主图（K线 + MA）与副图（Volume + MACD）的同步联动。
    3.  实现买卖信号 Markers 的渲染与点击交互。
    4.  开发左侧回测控制面板和底部绩效卡片。

### Sprint 4: 自选股、配置与联调 (Week 4)
*   **目标**：完成剩余视图，打通视图间的数据流转。
*   **任务**：
    1.  开发自选股视图（分组管理、Sparkline 迷你图）。
    2.  开发参数配置视图（复杂表单、脏检查、本地存储）。
    3.  **核心联调**：实现“选股结果 -> 加入自选 -> 回测视图查看”的全链路闭环。

### Sprint 5: 优化、测试与交付 (Week 5-6)
*   **目标**：提升性能，完善细节，达到生产可用标准。

## 十、前端全量计算架构（P4：K线形态 + 自编指标本地计算）(2026-06-23 设计)

> **章节性质**：P4 阶段方案设计，待实施。本章节记录"前端全量计算"架构的完整设计，包括 DataStore（IndexedDB 数据缓存）、计算引擎（Web Worker + 纯 TS 形态算法）、选股器适配，以及 Phase 1 原型验证计划。

### 10.1 架构决策背景与核心理念

#### 10.1.1 决策背景

原方案（后端 ETL 预计算所有形态 + 后端筛选）存在以下问题：
1. **非标准指标灵活性差**：K线形态、自编指标等非标准指标每次新增都要改后端 ETL + 数据库 schema + 同步逻辑，迭代慢。
2. **前后端耦合深**：每加一个指标都涉及"采集 → 计算 → 落库 → 同步 → 导出 → API → 前端"全链路改动。
3. **用户自编指标无法落地**：用户自定义公式不可能在后端动态执行（安全风险 + 性能问题）。

#### 10.1.2 核心理念

| 层级 | 职责 | 范围 |
|------|------|------|
| **后端** | 精准数据源 + 标准技术指标 | OHLCV 原始数据、MA/RSI/MACD/BOLL 等标准指标数值、基础财务字段 |
| **前端** | 非标准指标 + 自编指标 + 筛选逻辑 | 5 种 K 线形态、用户自编指标、选股筛选执行 |

**设计原则（K 2026-06-23 决策）**：
1. **零 WASM 依赖**：形态算法用纯 TypeScript 手写，绝不引入 WASM 编译工具链。
2. **零二进制依赖**：数据格式用 JSON + Gzip，绝不引入 Parquet/Arrow 等二进制解析库。
3. **架构极简**：后端彻底退化为"数据分发器"，前端掌握所有非标准指标的计算控制权。
4. **渐进式验证**：先 10 只股票 Mock 跑通闭环，再扩大到全市场 5000 只。

### 10.3 DataStore 数据缓存层（IndexedDB）

#### 10.3.1 数据模型

**数据库**：`quant_data` (v1)
- `DB_VERSION = 1`：每次修改数据结构时递增，触发 upgrade 清理旧缓存
- `DATA_VERSION = '1.0'`：数据格式版本，用于未来 migration 判断

| Object Store | Key Path | 字段 | 说明 |
|-------------|----------|------|------|
| `stocks_snapshot` | `code` | `code, name, listed_board, industry, trade_date, ohlcv, indicators` | 每只股票的 300 天 OHLCV + 最新交易日标准指标 |
| `meta` | `key` | `key, value` | 元数据（latest_trade_date, data_version, updated_at） |

**元数据 Key 命名规则（环境隔离）**：
```typescript
const META_KEY = `meta_${import.meta.env.MODE}`; // 'meta_development' / 'meta_production'
```
> 多环境（dev/staging/prod）共用 IndexedDB 时，通过环境前缀避免相互覆盖。

**MetaValue 类型定义**：
```typescript
interface MetaValue {
  latest_trade_date: string;  // YYYY-MM-DD
  version: string;            // 数据格式版本，用于未来 migration
  updated_at: number;         // Unix timestamp (秒)
}
```

**数据版本迁移策略**：
- 修改 `StockSnapshot` 或 `OHLCVItem` 类型结构（新增/删除字段）时，必须递增 `DB_VERSION`
- IndexedDB `upgrade` 回调中删除旧 ObjectStore 并重建，强制全量重新拉取
- 禁止在旧版本数据上做字段兼容，保持数据格式纯净

**idbStore 核心实现**：
```typescript
// src/lib/data/idbStore.ts
import { openDB, IDBPDatabase } from 'idb';

const DB_NAME = 'quant_data';
const DB_VERSION = 1;

export async function openQuantDB(): Promise<IDBPDatabase> {
  return openDB(DB_NAME, DB_VERSION, {
    upgrade(db, oldVersion) {
      if (oldVersion < DB_VERSION) {
        if (db.objectStoreNames.contains('stocks_snapshot')) {
          db.deleteObjectStore('stocks_snapshot');
        }
        if (db.objectStoreNames.contains('meta')) {
          db.deleteObjectStore('meta');
        }
      }
      db.createObjectStore('stocks_snapshot', { keyPath: 'code' });
      db.createObjectStore('meta', { keyPath: 'key' });
    },
  });
}
```

### 10.4 计算引擎层（Web Worker）

#### 10.4.1 核心架构原则：计算线程与 UI 线程内存隔离

**严禁主线程向 Worker 传递全量快照数据**（如 `StockSnapshot[]` 十几 MB 的 JS 对象）。`postMessage` 会触发浏览器结构化克隆（Structured Clone），导致主线程严重卡顿且内存瞬间翻倍。

**正确数据流**：
```
主线程                         Worker 线程
   │                               │
   │──── SCREEN 指令 ─────────────▶│   （只传筛选条件，几十字节）
   │                               │
   │                               │── fetch('/api/snapshot/all') ──▶ 后端
   │                               │   （Worker 内部直接拉取数据）
   │                               │
   │                               │── JSON.parse()
   │                               │   （Worker 内部解析，不占主线程）
   │                               │
   │                               │── IndexedDB 写入
   │                               │   （Worker 内部缓存管理）
   │                               │
   │                               │── 形态计算 + 筛选
   │                               │
   │◀── SCREEN_RESULT ────────────│   （只返回命中结果，通常几百条）
   │                               │
```

**优势**：
- 主线程零阻塞：数据拉取、解析、计算全部在 Worker 线程完成
- 内存零复制：数据始终在 Worker 内部，主线程只接收最终结果
- 架构清晰：Worker 自包含"数据获取 + 缓存 + 计算 + 筛选"完整链路

#### 10.4.2 5 种 K 线形态算法（纯 TS 实现）

**原则**：不依赖任何外部库，纯 TypeScript 实现 TA-Lib 等价逻辑。

| 形态 | 判定逻辑 | 所需K线数 | TA-Lib 对应函数 |
|------|---------|-----------|----------------|
| **锤子线 Hammer** | `实体处于上半部 && 下影 ≥ 实体×2 && 上影 < 实体×0.1` | 1 根 | `CDLHAMMER` |
| **看涨吞没 Bullish Engulfing** | `前日阴 && 今日阳 && 今日开 < 前日收 && 今日收 > 前日开` | 2 根 | `CDLENGULFING (+100)` |
| **看跌吞没 Bearish Engulfing** | `前日阳 && 今日阴 && 今日开 > 前日收 && 今日收 < 前日开` | 2 根 | `CDLENGULFING (-100)` |
| **早晨之星 Morning Star** | `Day1 大阴 && Day2 小实体(十字星)跳空低开 && Day3 大阳穿入Day1实体≥30%` | 3 根 | `CDLMORNINGSTAR` |
| **黄昏之星 Evening Star** | `Day1 大阳 && Day2 小实体(十字星)跳空高开 && Day3 大阴穿入Day1实体≥30%` | 3 根 | `CDLEVENINGSTAR` |

**算法参数（可调）**：
- 穿透系数（penetration）：0.3（晨星/昏星的 Day3 需穿入 Day1 实体 30% 以上）
- 实体大小阈值：`abs(close - open) / open ≥ 0.01`（大实体判定）
- 十字星判定：`abs(close - open) / open < 0.003`

#### 10.4.3 Worker 消息协议

```typescript
// 主线程 → Worker（指令类，体积极小，仅几十字节）
type WorkerRequest =
  | { type: 'LOAD_DATA'; force?: boolean }      // 触发数据加载（首次/强制刷新）
  | { type: 'SCREEN'; id: string; conditions: ScreenCondition[] }
  | { type: 'CANCEL'; id: string };

// Worker → 主线程
type WorkerResponse =
  | { type: 'LOAD_PROGRESS'; progress: number; phase: 'fetch' | 'parse' | 'cache' }  // 0-100
  | { type: 'LOAD_DONE'; count: number; latest_trade_date: string }
  | { type: 'LOAD_ERROR'; message: string }
  | { type: 'SCREEN_PROGRESS'; id: string; progress: number } // 0-100
  | { type: 'SCREEN_RESULT'; id: string; result: ScreenResultItem[] }
  | { type: 'SCREEN_ERROR'; id: string; message: string };

interface ScreenCondition {
  fieldKey: string;              // pattern_hammer / rsi_6 / ...
  op: 'AND' | 'OR' | 'NOT';
  operator?: '>' | '>=' | '<' | '<=' | '==' | 'cross_up' | 'cross_down';
  value?: number;
  lookbackDays?: number;         // 仅形态类使用
}

interface ScreenResultItem {
  code: string;
  name: string;
  // 基础字段（直接从 snapshot 取）
  close: number;
  change_pct: number;
  market_cap: number;
  turnover_rate: number;
  // 命中的形态列表（用于结果表格 tag 展示）
  matchedPatterns: string[];
}
```

**LOAD_DATA 触发时机**：
- 选股页面首次挂载时
- 用户点击"刷新数据"按钮时
- 每日首次打开应用时（对比本地缓存日期）

**SCREEN 触发时机**：
- 用户点击"开始选股"按钮
- Worker 内已缓存数据则直接计算，否则自动先 LOAD_DATA 再计算

**数据持有模式**：Worker 内部持有全量快照数据的内存引用（`Map<code, StockSnapshot>`），LOAD_DONE 后驻留内存，后续 SCREEN 直接复用。

#### 10.4.4 Worker 管理策略

| 策略 | 实现方式 |
|------|---------|
| **视图级生命周期** | Worker 在 `StockPickerView` 挂载时创建、卸载时销毁（`worker.terminate()`）。**不提升到 App 全局**，避免用户切换到其他视图时 Worker 仍占用内存。 |
| **单例 Worker** | 选股页内单例，避免重复创建导致资源竞争 |
| **任务取消** | 新任务到达时，若上一任务未完成，发 `CANCEL` 消息 + 重新初始化计算状态 |
| **分片计算** | 每 100 只股票计算后 `setTimeout(0)` 让出 Worker 线程，避免长时间独占 |
| **进度回调** | 每处理 500 只（或 10%）返回一次进度 |
| **内存隔离** | 全量数据始终在 Worker 内部，主线程只接收筛选结果（通常几百条），零结构化克隆开销 |
| **数据拉取** | Worker 内部直接 `fetch('/api/snapshot/all')`，JSON.parse 在 Worker 线程完成，主线程零阻塞 |

**useScreenWorker 生命周期实现**：
```typescript
// src/hooks/useScreenWorker.ts
import { useEffect, useRef, useState } from 'react';

export function useScreenWorker() {
  const workerRef = useRef<Worker | null>(null);

  useEffect(() => {
    const worker = new Worker(
      new URL('../workers/screen.worker.ts', import.meta.url),
      { type: 'module' }
    );
    workerRef.current = worker;

    return () => {
      worker.terminate();
      workerRef.current = null;
    };
  }, []); // 空依赖，仅挂载/卸载时触发

  // ... screen / cancel / loadData 方法
}
```

### 10.6 标准指标处理（后端计算 + 前端比较）

#### 10.6.1 职责划分

| 指标类型 | 计算位置 | 前端操作 |
|---------|---------|---------|

### 10.12 测试覆盖率强制要求

形态算法单元测试**必须达到 100% 分支覆盖率**（Vitest 输出 `100% Statements / 100% Branches / 100% Functions / 100% Lines`）。

**必须覆盖的边界值测试场景**：

| 测试类别 | 具体场景 |
|---------|---------|
| **十字星/一字板** | 实体为 0（open == close）的极端 K 线 |
| **涨跌停板** | 涨停一字板（open=high=low=close，无实体、无下影、无上影） |
| **停牌/缺失数据** | 单日 volume == 0 或数据为 NaN/null 的情况 |
| **临界穿透阈值** | 穿透系数恰好等于 30%（边界值） |
| **数据不足** | 只有 1-2 根 K 线时，需要 3 根 K 线的形态（晨星/昏星）应返回 false |
| **方向混淆** | 锤子线与倒锤子线的方向判定（上影 vs 下影） |
| **吞没边界** | 实体大小完全相等（open-close == 0，恰好吞没） |

测试文件：`tests/lib/indicators/patternDetector.test.ts`

### 10.13 风险与应对

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| 首次加载数据量大（1-1.5MB Gzip） | 低 | Loading 骨架屏 + 进度条 + 友好提示文案 |
| Worker 计算 5000 只股票耗时 | 中 | 分片计算 + 进度回调 + Web Worker 不阻塞 UI |
| 形态算法与 TA-Lib 结果不一致 | 中 | Phase 1 用 10 只真实股票人工核对；后续用 TA-Lib Python 版做批量对比测试 |
| IndexedDB 浏览器兼容性 | 低 | 所有主流浏览器均支持；降级方案：内存存储（刷新后重拉） |
| 自编指标安全问题 | 低 | Phase 1-3 不开放自由输入，用预置配置方式；远期引入 AST 解析器 |
| 形态算法边界 case 遗漏 | 中 | 单元测试 100% 分支覆盖率强制要求；含十字星 / 一字板 / 停牌等极端 K 线全覆盖 |
