## 一、模块定位

将选股器（StockPicker）中用户配置的筛选条件转换为组合回测策略，在历史数据上模拟调仓过程，输出绩效指标和可视化报告。

**与现有模块的关系**：
- 完全独立于 `backtest/`（单股回测 V1），不共享代码和状态
- 依赖 `indicatorConfig` 中的指标参数配置（MACD/RSI/MA/BOLL/KDJ 参数）
- 通过 `FilterNode` AST 从 `stock-picker` 获取选股条件

### 3.0.1 竞态条件防护（K审阅V3#1）

**问题**：`Loading` 阶段发起的网络请求可能在组件卸载或取消后依然触发 `setState`，导致过期响应覆盖新结果。

**对策**：引入 **Session ID + AbortController** 绑定机制。

```typescript
// 回测会话管理器
class BacktestSessionManager {
  private sessionId = 0;         // 单调递增的会话 ID
  private abortController: AbortController | null = null;
  private worker: Worker | null = null;

  /** 启动新回测，任何旧会话自动失效 */
  start(): BacktestSession {
    this.sessionId++;
    this.abortController?.abort();  // 取消旧的网络请求
    this.worker?.terminate();       // 终止旧的 Worker
    this.worker = null;
    const currentId = this.sessionId;
    return {
      id: currentId,
      signal: this.abortController!.signal,
      /** 所有异步回调必须先校验此 ID 是否仍为当前激活会话 */
      isActive: () => this.sessionId === currentId,
    };
  }
}
```

**使用规则**：
- 所有 `fetch` 调用传入 `{ signal: session.signal }`，取消时自动中止
- 所有 `setState` 回调前先调用 `session.isActive()`，若不匹配则静默丢弃
- Worker 的 `onmessage` 回调同样校验 Session ID
- 取消流程：`AbortController.abort()` → `Worker.terminate()` → 状态机回到 Idle

### 3.0.2 Worker 生命周期管理器（K审阅V3#2，Phase 2 迭代）

**问题**：Worker 被 `terminate()` 后，其内部持有的 `ArrayBuffer` 等可转移对象在主线程中可能未被显式释放，高频取消-重试操作导致内存累积。

**Phase 2 对策**（V3 中预留接口，不阻塞 Phase 1）：
- 建立 `WorkerLifecycleManager`，在 `terminate()` 后强制解除所有 `MessagePort` 引用
- 在 Worker 内部 `try...finally` 中释放 `Transferable` 对象
- 在主线程侧对 `onmessage` 引用置 `null`，暗示 GC 回收
- UI 中的取消操作日志增加 `WorkerLifecycleManager` 的内存释放记录

### 4.3 选股条件传递方式

**选股器 → 策略回测页面**：通过 URL 参数传递 `FilterNode` AST 的 JSON 序列化（压缩后通常 < 2KB，适合 URL）：

```
/picker → /strategy-backtest?tree=eyJ0eXBlIjoiYW5kIiwiY2hpbGRyZW4iOlt7...
```

**读取方式**：`useSearchParams().get('tree')` → `JSON.parse(atob(tree))` → 直接喂给引擎。

### 4.4 URL 安全校验（K审阅#7）

对 `tree` 参数施加严格的安全校验，防止解析炸弹（Billion Laughs Attack）和 DoS：

```typescript
const TREE_MAX_SIZE = 2048;      // 2KB 大小上限
const TREE_MAX_DEPTH = 10;       // 递归深度上限

function parseTreeParam(raw: string | null): FilterNode | null {
  if (!raw) return null;
  // 1. 大小校验
  if (raw.length > TREE_MAX_SIZE) {
    throw new Error('选股条件过长，请简化后重试');
  }
  // 2. Base64 解码
  const decoded = atob(raw);
  if (decoded.length > TREE_MAX_SIZE) {
    throw new Error('选股条件过长，请简化后重试');
  }
  // 3. JSON 解析
  const parsed = JSON.parse(decoded);
  // 4. 递归深度校验 + Schema 校验
  validateFilterNode(parsed, 0);
  return parsed;
}
```

### 4.5 导航原子性契约（K审阅#8）

**单向数据流契约**：选股器为唯一数据源（Source of Truth）。

- 回测页面**仅消费** `tree` 参数，禁止回写或修改
- 点击"修改选股条件"回跳时，携带**原始不变的 `tree` 参数**回选股器
- 若用户在回测页面修改了 URL 参数后点击"修改选股条件"，触发确认弹窗：
  > ⚠ 当前回测条件与选股器条件可能不一致，是否跳转回选股器重新配置？

### 5.1 防腐层（Anti-Corruption Layer）（K审阅#10）

在引擎外围引入适配器层，隔离数据格式差异：

```
┌──────────────────────────────┐
│        UI 组件层              │
├──────────────────────────────┤
│    防腐层（适配器/转换器）     │  ← 数据格式转换在此层完成
├──────────────────────────────┤
│   引擎核心（types/engine）    │  ← 定义清晰的 Input/Output 契约
├──────────────────────────────┤
│    Worker / 数据加载          │
└──────────────────────────────┘
```

**具体实现**：
- `engine.ts` 定义 `EngineInput` 和 `EngineOutput` 接口作为契约
- `dataLoader.ts` 在防腐层中将后端 API 响应格式转换为 `EngineInput`
- `filterTreeAdapter.ts` 在防腐层中将选股器 AST 转换为引擎可执行的 `FilterNode`
- 未来引擎重构时，仅需调整适配器，不影响 UI 层

#### 5.2.1 配置迁移器（Migration Runner）（K审阅V3#4）

**问题**：旧版 `StrategyBacktestDefaultsPanel.tsx` 使用孤立键存储（如 `backtest-defaults`），切换到新命名空间后用户自定义参数将丢失。

**对策**：在模块初始化时自动执行渐进式配置迁移。

```typescript
// storage.ts — 配置迁移器
const OLD_KEYS = ['backtest-defaults', 'backtest-defaults-v2'];  // 旧版存储键
const NEW_PREFIX = 'strategy-backtest/';

function migrateConfig(): void {
  for (const oldKey of OLD_KEYS) {
    const oldValue = localStorage.getItem(oldKey);
    if (oldValue === null) continue;
    try {
      // 1. 解析旧值
      const oldConfig = JSON.parse(oldValue);
      // 2. 映射至新结构（字段名变更时做转换）
      const newConfig = mapOldToNew(oldConfig);
      // 3. 写入新命名空间
      localStorage.setItem(`${NEW_PREFIX}defaults`, JSON.stringify(newConfig));
      // 4. 删除旧键
      localStorage.removeItem(oldKey);
      // 5. 记录迁移日志（供回滚参考）
      console.info(`[ConfigMigration] ${oldKey} → ${NEW_PREFIX}defaults (OK)`);
    } catch (e) {
      console.warn(`[ConfigMigration] ${oldKey} 迁移失败，保留旧值`, e);
    }
  }
}
```

#### 5.2.2 配置语义隔离（K审阅V3#8）

**问题**：组合回测滑点（按成交额百分比计算，反映流动性冲击）与单股回测滑点（固定万分比，反映交易成本）业务语义不同，共用同一输入字段导致用户混淆。

**对策**：
- 在 UI 标签中显式补充场景附属说明：
  - 策略回测侧："组合滑点（按成交额百分比计算）"
  - 单股回测侧："个股滑点（固定万分比）"
- 即便底层类型结构相同，也各自独立实现输入组件，不对未来业务分化做假设
- 存储层通过不同命名空间保证实例隔离

## 六、数据加载流程

### 6.1 数据依赖

| 数据 | 来源 | 说明 |
|------|------|------|
| 全市场 OHLCV | `GET /api/snapshot/all` | Parquet 缓存，~1M 行 |
| 股票快照 | `GET /api/snapshot/all` | is_st、市值、板块 |
| 交易日历 | `GET /api/trade-calendar` 或 `trade_dates` 字段 | **P0 前置依赖**，缺失时阻断回测 |
| 基准指数 | `GET /api/kline/000300.SH` | 沪深300日线，可选 |
| 基本面数据 | `stockDailySnapshot` | **仅最新交易日快照，不用于历史回测筛选** |

### 6.2 数据准确性与校验

#### 6.2.1 基本面前瞻偏差阻断（K审阅#1）

**问题**：基本面指标（PE/PB/市值/换手率）使用最新交易日快照，若用于回测筛选条件，会引入前瞻性偏差（Look-ahead Bias）。

**对策**：
- `dataLoader.ts` 加载完成后，检测 `FilterNode` 中是否包含 `fundamental` 类字段（`pe`、`pe_ttm`、`pb`、`market_cap`、`turnover_rate`、`vol_ratio_5`）
- 若检测到，在回测启动前**阻断**，弹出明确错误提示：
  > ⚠ 当前选股条件包含基本面指标（PE/PB/市值等），回测引擎无法获取历史时点的基本面数据。请移除这些条件后重试，仅保留技术指标（MA/MACD/RSI/BOLL）和行情数据（价格/涨跌幅/成交量）等可基于OHLCV计算的指标。
- 后续可迭代：当后端提供历史基本面快照时，移除该阻断

#### 6.2.2 数据缺失断言（K审阅#3）

**废除兜底逻辑**，改为强制校验：

```typescript
// dataLoader.ts — 数据加载后校验
function validateDataIntegrity(
  allOhlcv: Map<string, number[][]>,
  snapshots: Map<string, StockSnapshot>,
  tradeDates: string[]
): string[] {
  const errors: string[] = [];
  // 1. 交易日历不得为空
  if (!tradeDates || tradeDates.length === 0) {
    errors.push('交易日历为空，无法确定回测区间');
  }
  // 2. OHLCV 必须包含 pre_close 列（索引6）
  for (const [code, bars] of allOhlcv) {
    if (bars.length > 0 && bars[0].length < 7) {
      errors.push(`${code}: OHLCV 缺少 pre_close 列`);
      break;
    }
  }
  // 3. snapshot 必须包含 is_st 字段
  for (const [code, snap] of snapshots) {
    if (snap.isSt === undefined) {
      errors.push(`${code}: snapshot 缺少 is_st 字段`);
      break;
    }
  }
  return errors;
}
```

校验失败时，UI 显示红色 Alert 并阻止回测启动。

#### 6.2.3 错误分类与分级处理（K审阅V3#7）

将数据错误分为两类，差异化 UI 处理：

| 类型 | 条件 | UI 行为 | 示例 |
|------|------|---------|------|
| **硬错误（硬阻断）** | 系统性数据缺失，影响全局 | 红色 Alert 阻止回测，显示错误摘要 | 全量 OHLCV 无 `pre_close` 列、交易日历为空 |
| **软错误（警告+部分降级）** | 个别股票/个别日期的数据问题 | 黄色 Alert 允许"忽略警告并继续"，回测报告中标注受影响股票 | 个别停牌股数据缺失、某只股票缺少 `is_st` |

**软错误处理流程**：
```
1. 检测到软错误（如股票 "000001.SZ" 缺少 is_st）
2. 在 UI 中展示可展开的详情列表：["000001.SZ: 缺少 is_st，停牌/退市处理可能不准确"]
3. 提供"忽略警告并继续"按钮（默认不启用，需用户确认）
4. 用户确认后，引擎将该股票标记为"非ST"（保守处理）
5. 回测报告中显著标注"受影响的股票和日期区间"
```

#### 6.2.4 基准指数对齐规则（K审阅V3#6）

**问题**：基准指数（沪深300）与策略持仓的交易日可能不一致（指数休市日个股有交易），导致 Alpha/Beta/信息比率等指标计算产生系统性偏差。

**对齐规则**：
1. 在绩效指标计算前，强制将策略净值与基准数据按**交易日历主表**进行左连接对齐（Left Join）
2. 缺失的基准点采用**前向填充**（Forward Fill）：用上一个已知基准值填充缺失点
3. 生成数据质量元数据，随指标一同展示：
   - `基准缺失率: 0.5%`（缺失天数 / 总交易日）
   - `对齐方式: 前向填充`
4. 若基准缺失率 > 5%，在指标区域显示黄色警告："基准缺失率较高（5.2%），Alpha/Beta 等指标参考价值有限"

### 6.3 条件下推（Predicate Pushdown）— 双阶段策略（K审阅#4, V3#3）

**目标**：不加载全市场 OHLCV 数据，将选股条件前置传递给后端，仅拉取候选股票池的历史数据。

**核心挑战**：当 `FilterNode` AST 包含嵌套 `OR` 逻辑（如 `(A AND B) OR (C AND D)`）时，强行扁平化为单一范围参数会导致逻辑失真。

**对策**：采用 **"后端粗筛 + 前端预过滤"双阶段策略**。

#### 阶段一：后端粗筛（提取无歧义条件）

从 `FilterNode` AST 中提取**绝对无歧义的筛选条件**传给后端 API：

| 可提取条件 | 提取方式 | 示例 |
|-----------|---------|------|
| 板块枚举（`market` 节点） | 直接提取 `boards` 数组 | `listed_board=主板,创业板` |
| 市值绝对值（`range` 节点中 `field=market_cap`） | 提取 `min`/`max` | `market_cap_min=50e8&market_cap_max=500e8` |
| 与根节点直接 `AND` 连接的简单 `range` | 从扁平 `AND` 子句中提取 | `price_min=5&price_max=50` |

**不可提取（标记为"引擎侧过滤"）**：
- 嵌套在 `OR` 或 `NOT` 中的条件
- 跨指标复合条件（如 `(pe<10 AND pb<1) OR (market_cap>1000e8)`）
- 技术形态（`pattern`）和 K 线形态（`kline`）节点

#### 阶段二：引擎预过滤

后端返回初选池后，`filterTreeAdapter` 在引擎内用**完整的 `FilterNode` AST** 对候选池进行二次精准过滤，确保 AST 逻辑完全忠实还原：

```typescript
// dataLoader.ts — 双阶段加载
async function loadData(tree: FilterNode): Promise<EngineInput> {
  // 阶段一：提取可下推条件
  const pushdown = extractPushdownPredicates(tree);
  // 调用后端 API 获取初选池
  const candidateCodes = await fetch(`/api/stocks/?${pushdown.toQueryString()}`);
  // 阶段二：拉取初选池的完整 OHLCV
  const ohlcv = await fetch(`/api/snapshot/all?codes=${candidateCodes.join(',')}`);
  // 引擎侧用完整 AST 再过滤
  const engineInput = buildEngineInput(ohlcv, tree);
  return engineInput;
}
```

**兜底**：若后端不支持 `?codes=` 过滤，降级为全量加载，但引擎侧过滤逻辑不变。

### 6.5 IndexedDB 批次缓存（断点续传）（K审阅V3#5）

**目标**：回测过程中浏览器意外关闭或网络断连后，支持从断点恢复，无需从头重跑。

**实现方式**：
- 每个批次完成时，将中间净值序列持久化到 `IndexedDB`
- 缓存键 = `{回测参数Hash}:{批次号}`，值 = `{净值序列, 时间戳}`
- 重启时检测缓存有效性：
  - 校验 `时间戳`（超过 24 小时视为过期）
  - 校验 `参数Hash`（参数变更时缓存失效）
- 有效缓存 → 从断点恢复，跳过已完成的批次
- 无效缓存 → 从头开始，自动清除旧缓存

**相册数据大小**：通常 200 只 × 500 天 ≈ 100KB/批次，总缓存 < 2MB。

## 十二、风险和依赖

| 风险/依赖 | 影响 | 缓解措施 |
|-----------|------|---------|
| 后端未提供 `pre_close`/`is_st` 列（P0） | 涨跌停判定、收益率计算错误 | **阻断回测**，`validateDataIntegrity` 抛出明确错误 |
| 后端未提供交易日历（P0） | 回测区间完整性、夏普/回撤计算基准错误 | **阻断回测**，UI 提示"交易日历缺失，请联系后端确认" |
| 后端不支持 `/api/snapshot/all?codes=` 过滤 | 全量加载 5000+ 股票 OHLCV，内存 ~300MB | Phase 1 降级全量加载，Phase 2 推进后端支持 |
| `filterTreeAdapter.ts` 选股器结构变更 | 转换逻辑断裂，AST 解析失败 | 采用 Visitor 模式 + 契约测试捕获 Schema 变更 |
| 基本面前瞻偏差 | 回测绩效严重失真 | 检测到基本面字段时阻断，提示用户移除 |
| URL 参数超长或深层嵌套 | 解析炸弹 DoS 攻击 | 2KB 大小上限 + 10 层递归深度 + Schema 预检 |
| 回测过程中浏览器崩溃 | 已完成的批次计算结果丢失 | **IndexedDB 批次缓存（Phase 1）**：每个批次完成时持久化中间净值序列，重启时校验缓存有效性（时间戳+参数 Hash），支持断点恢复 |

## 十三、边际优化（Phase 2/3 迭代，K审阅第三轮）

以下 7 条建议均属 P2 级，不阻塞 Phase 1 开发，在后续迭代中按优先级逐步完善。

### 13.1 状态机"重置"拆分（V3.1#1）

当前设计：`Success/Failed → Idle` 触发"重置"动作，语义模糊。

**Phase 2 改进**：将"重置"拆分为两个独立按钮：
- **"清空结果"**（仅隐藏图表，保留所有参数，立即回到 Idle）
- **"恢复默认设置"**（参数重置为系统默认值，同时清空结果）

或在"重置"时弹出轻量确认对话框，让用户勾选"是否同时重置参数"。

### 13.2 URL tree 与 Storage 职责分层（V3.1#2）

当前设计：`tree` 参数携带选股逻辑，执行参数从 Storage 读取，但未正式声明契约。

**Phase 2 改进**：明确职责分层契约：
- `tree` 参数**仅承载选股逻辑**（FilterNode AST），不包含任何策略执行参数（资金/费用/风控/调仓频率）
- 执行参数一律从 Storage 读取
- 若 `tree` 中意外包含执行参数，在解析时主动剥离并告警
- 将契约写入 `filterTreeAdapter.ts` 的类型定义注释中

### 13.3 筛选审计报告（Filter Audit Trail）（V3.1#3）

当前设计：双阶段过滤过程不透明，排查问题困难。

**Phase 2 改进**：在 `dataLoader` 中生成筛选审计报告，作为回测结果元数据的一部分：
- 下推至后端 API 的具体条件列表（Query String 展示）
- 引擎侧二次过滤前/后的股票数量对比
- 被两次过滤剔除的代表性股票及其剔除原因
- 在 UI 中以折叠面板形式供高级用户审阅

### 13.4 缓存键 Hash 规范公式（V3.1#4）

当前设计：`IndexedDB` 缓存键的 `参数Hash` 计算范围未定义。

**Phase 2 改进**：明确 Hash 包含以下字段：
- 选股条件 AST 的序列化字符串
- 回测区间起始/结束日期
- 调仓频率
- 最大持仓数量
- 风控开关状态
- **不包含**：手续费/滑点等成本参数（不影响交易信号，仅影响收益率数值）

在缓存读写接口中统一封装 Hash 计算逻辑，并记录 Hash 版本号。

### 13.5 基本面字段分级阻断（V3.1#5）

当前设计：检测到任何基本面字段即阻断回测，属于全有/全无的硬阻断。

**Phase 2 改进**：将基本面指标按前瞻性风险等级分类：

| 等级 | 字段 | 阻断策略 |
|------|------|---------|
| **高风险（硬阻断）** | PE、PB、市值、换手率、成交量比 | 阻断，提示"需要历史估值数据" |
| **低风险（允许通过）** | 板块归属、ST 状态 | 允许通过，引擎用历史快照回溯 |

### 13.6 迁移回滚安全网（V3.1#6）

当前设计：`migrateConfig` 为一次性不可逆操作（读取→写入→删除）。

**Phase 2 改进**：增加回滚安全网：
- 迁移前将旧配置完整备份至 `backup/` 命名空间（保留时间戳）
- 不立即删除旧键，迁移成功后等待一个"稳定期"（7 天）或用户首次成功运行回测后再异步删除
- 提供"撤销迁移"入口（仅在出现异常时可见），从备份中还原配置

### 13.7 Schema 快照双源校验（V3.1#7）

当前设计：CI 每日自动拉取后端开发环境响应并更新快照，开发环境不稳定时产生噪音。

**Phase 2 改进**：改为"双源校验 + 人工确认"：
- CI 定时任务生成快照后，先与上一版本快照进行 Diff 对比
- 若发生变化，**不自动提交**，而是触发 PR（含 Diff 内容），由前端负责人审阅变更是否合理后方合并
- 紧急情况下，允许通过 `--force-update-snapshot` 参数跳过人工确认
